## LANDSCAPE + LOOP REVIEW
The motion I chose is inbound-to-evangelist conversion, specifically turning free users who have already generated a diagram into paid subscribers through automated behavioral detection and personalized outreach.
I ruled out three obvious motions first. Cold programmatic SEO content was the first. Tools like Jasper, Writesonic, and every AI content farm already do this. The problem is it takes 90 days minimum to see organic ranking results and diagrams.so needs paid users now, not traffic. I also ruled out comparison content engines because they require ongoing maintenance as competitors change their features and carry brand risk if claims go stale. I ruled out social media automation loops because engineers do not convert from brand social posts, they convert from seeing the tool solve their specific problem.
The motion I chose is fundamentally different from all three because the leads already exist inside the product. Every free user who generates a diagram is a warmer lead than anything instantly.ai or Apollo can source from a database of 450 million contacts, because they have already experienced the value. The job to be done for the operator is converting those warm leads systematically rather than leaving them to churn silently.
The insight from my own experience scaling Basics of Sign Language to 120,000 users is relevant here. The growth did not come from content or ads. It came from a simple popup that asked users for feedback at the moment they had just experienced value. That moment of peak engagement is when conversion happens. For diagrams.so that moment is immediately after a user generates their first diagram and sees it rendered. Everything in this system is designed around that moment.

## README
What I built: A fully local automated conversion pipeline for diagrams.so that detects behavioral signals from free users, infers their professional context using a local LLM, routes them into a personalized three-step email nurture sequence, and surfaces AI-powered upgrade suggestions at the exact moment they hit the free tier limit. A review dashboard gives the operator visibility and approval control over every item before anything external is triggered.
Who it is for: The operator at diagrams.so who wants to convert existing free users to paid without hiring a sales team or running ads. The end user is a professional engineer, platform engineer, or solutions architect who generated a diagram to solve a real production problem and needs to understand what the paid tier unlocks for their specific use case.
The job to be done: Free users who generate diagrams are the highest intent leads diagrams.so already has. Right now nothing systematic happens after they generate. This system closes that gap automatically.
Why this approach over alternatives: Cold outreach tools like instantly.ai work by finding leads from external databases and sending volume. This system uses product behavior as the signal instead, which means every outreach is contextually relevant to what the user actually built. A Platform Engineer who generated a HIPAA pipeline diagram gets a different email than a Solutions Architect who generated a zero trust architecture. The personalization is not template-based, it is derived from the prompt itself via Ollama inference.
How to run it: Prerequisites are Python 3.11 plus, Node.js 20 plus, and Ollama running locally with phi3:mini pulled. Set NOTIFY_EMAIL, NOTIFY_GMAIL_APP_PASSWORD, DEMO_MODE=true, and optionally NEXT_PUBLIC_POSTHOG_KEY in dashboard/.env.local. Run the Python pipeline with python run.py from the pipeline folder to generate gallery items. Run npm run dev from the dashboard folder for the operator interface. The Demo Mode button on /generate loads pre-built examples instantly without Ollama running.
Tools and models used: Ollama with phi3:mini for local LLM inference, diagrams.net embed for rendering, Next.js 16 for the dashboard and API layer, Python 3.11 for the gallery pipeline, pytrends for keyword discovery, Nodemailer with Gmail for email sending, PostHog for event tracking and funnel analytics, node-cron pattern via sequence-runner.ts for email scheduling, Claude and Cursor for development assistance.
Sample run: Type "HIPAA compliant healthcare data pipeline on AWS" in the generate page or click Demo Mode and select HIPAA Fintech. The diagram renders with RDS, KMS, S3, CloudTrail, and Glue in swimlane tiers. Two pro upgrade suggestions appear: audit logging automation and compliance report generation. Saving triggers the sequence. Email one arrives immediately with the diagram link and upgrade suggestions addressed to a Platform Engineer at a Fintech. Email two arrives at day three with expanded production rationale. Email three arrives at day seven with the free tier limit warning and upgrade CTA. In demo mode with DEMO_MODE=true all three arrive within two minutes.
Brand safety and quality gates: Nothing publishes externally without a human clicking Approve in the dashboard. Social post drafts, gallery metadata, and email content all sit in the review queue. The operator reads every item before it moves to approved. The notify route sends only to the operator inbox during the prototype, never to real users. The README documents this boundary explicitly. LLM output passes through a sanitization layer that strips placeholder text before any content is saved or displayed.
Known limitations: No public diagrams.so API exists so the prototype cannot make real calls to the product. The rendering uses the same diagrams.net engine that diagrams.so is built on as a proxy. phi3:mini produces simpler diagram specs than larger models. The intent scoring uses localStorage UUIDs which do not persist across devices. The email sequence sends to the operator inbox only. No authentication on the dashboard.
What I would do next: Integrate with the diagrams.so API when it ships, the integration points are documented. Switch to a larger hosted model for richer diagram specs. Add webhook support so the nurture sequence triggers from real diagrams.so signups rather than local saves. Build the usage-gated regeneration feature directly into diagrams.so's UI so the upgrade prompt appears in-product not just in email.
What I would cut if scope halved: Cut the gallery SEO pipeline and the social post generation entirely. The core value is the nurture loop and the conversion gate. A human copywriter can write social posts in five minutes once the diagram exists. The SEO loop takes months to show results. The nurture loop works immediately on existing users which is what diagrams.so needs.
